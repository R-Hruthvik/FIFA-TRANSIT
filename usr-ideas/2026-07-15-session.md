### Features
- No user profiles, admin profiles, or auth system has been built yet. The auth layer should support two methods: Google OAuth and email/password with robust protection (hashing, rate limiting, etc.). On Google sign-in, collect email and name from Google, but don't auto-fill the app username from the Google name — let the user pick any display name (anonymous or named). Internally assign a unique user ID (UUID) to each user so duplicate display usernames are allowed. User IDs are never shown on main pages — only visible in a dedicated Settings page.
- Admins use the same login page as regular users but authenticate exclusively via email/password (no Google auth). Admin credentials require strict, unique usernames and passwords with enhanced security features: daily password rotation, complex username requirements, and 2FA. After login, admins get access to a developer settings panel. Staff profiles are separate from admin — the auth system needs to support three roles: fan (Google or email/pass), staff (email/pass only), and admin (email/pass only with elevated security).

### Bugs
- Staff registration is strict: staff must provide a staff ID issued by the stadium department to sign up. Once the registration application is submitted, it goes into a pending/waiting list. An admin must approve it by cross-referencing the provided details against valid staff records. Staff can log in using any auth method (Google or email/password) once approved. This means the auth system needs a registration flow with role selection, staff ID verification, and an admin approval queue.
