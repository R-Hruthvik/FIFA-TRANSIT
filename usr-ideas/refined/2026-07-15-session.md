## Idea 1: Realistic Stadium Blueprint

**Priority:** medium

**Effort:** medium

**Category:** design

**Raw quote:**
> I want real stadium miniature blueprint instead of square and 4 circles as stadium you made in staff page real stadium 2d blueprint will be good, i have more

**Refined description:**
Replace the abstract square-with-4-circles stadium representation in the Staff Hub heatmap (`src/components/heatmap/StadiumSVG.tsx`) with a realistic 2D stadium blueprint — an oval bowl shape with stands, field markings, and actual gate positions on the perimeter. Gates should sit at real entrance points around the oval rather than at the four corners of a square. This makes the congestion visualization immediately readable as a real venue.

**Action items:**
- [ ] Design SVG oval stadium shape with stands and field markings
- [ ] Reposition gates to realistic perimeter locations
- [ ] Update heatmap color mapping to work with new stadium shape
- [ ] Ensure all 3 heatmap variants (StadiumSVG, ThermalGrid, StadiumMap) stay consistent

**Dependencies:**
- None — self-contained visual change in StaffHub

## Idea 2: Auth System with Google OAuth + Email/Password

**Priority:** high

**Effort:** large

**Category:** feature

**Raw quote:**
> We did not touch user profiles admin profiles and auths, so first auth is google auth or simple email and pass auth with robust protection, and we collect username and emailfrom google, and for app's username it will not be default selected same username from google, but let user type some anonymous or any username, we will have provide user id for each user so duplicate usernames is fine, we will not show user ids at any main pages we will only show it in settings page

**Refined description:**
Build the foundational auth layer with two methods: Google OAuth and email/password with robust protection (bcrypt hashing, rate limiting, secure session management). On Google sign-in, collect email and name but do NOT auto-fill the display username — let the user choose any name (anonymous or named). Assign each user a UUID internally so duplicate display usernames are allowed. User IDs are hidden from all main pages and only visible in Settings.

**Action items:**
- [ ] Set up NextAuth.js or similar auth framework
- [ ] Implement Google OAuth provider
- [ ] Implement email/password registration and login with hashing
- [ ] Add rate limiting on auth endpoints
- [ ] Create sign-up flow with custom username input (not pre-filled from Google)
- [ ] Add UUID generation per user
- [ ] Build Settings page showing user ID
- [ ] Ensure user IDs are excluded from all main page data fetches

**Dependencies:**
- Requires database (D1 or similar) for user storage
- Blocks: all role-based features (admin, staff)

## Idea 3: Admin Auth with Enhanced Security + Developer Settings

**Priority:** high

**Effort:** large

**Category:** feature

**Raw quote:**
> And for admin profile let admin will login with same way as any other user but with strict username and password which is unique and we can introduce many security features for admin logins like new pass for each days or complex username or 2fa and after admin logs in, we can give some developer settings to admin, also we didn't talk about staff profiles i mean admin will log in as same login page as other users but uses email and pass option but types username and password which is very secure and robust and does not user google auth

**Refined description:**
Admins authenticate via the same login page but use email/password only (no Google). Admin accounts have elevated security requirements: strict unique usernames, daily password rotation, complexity rules, and 2FA. After login, admins access a developer settings panel for configuration. This establishes three distinct roles: fan, staff, and admin.

**Action items:**
- [ ] Add role system to user schema (fan/staff/admin)
- [ ] Build admin-specific auth flow (email/pass only, no Google)
- [ ] Implement password complexity validation for admin accounts
- [ ] Add daily password rotation mechanism
- [ ] Add 2FA (TOTP or similar) for admin login
- [ ] Create admin approval flow for staff registrations
- [ ] Build developer settings page for admins
- [ ] Add role-based route guards

**Dependencies:**
- Requires Idea 2 (base auth system) to be built first
- Blocks: staff approval queue (Idea 4)

## Idea 4: Staff Registration with Approval Queue

**Priority:** high

**Effort:** medium

**Category:** feature

**Raw quote:**
> For staff profile staff can login as other users using any method but sign up or registration is very strict staff uses staff id issued by stadium department and when sign up or registration application is done it is went to waiting list to be approved by admin, admin should verify it by cross reference with any valid data on staffs

**Refined description:**
Staff registration requires a valid staff ID issued by the stadium department. Submitted applications go to a pending queue. Admins review and approve by cross-referencing against valid staff records. Once approved, staff can log in via any auth method (Google or email/password).

**Action items:**
- [ ] Create staff registration form with staff ID field
- [ ] Build pending/waiting list UI for admin review
- [ ] Add admin verification flow (cross-reference against staff records)
- [ ] Send notification to staff on approval/rejection
- [ ] Gate staff dashboard access behind approval status

**Dependencies:**
- Requires Idea 2 (base auth) and Idea 3 (admin auth)
